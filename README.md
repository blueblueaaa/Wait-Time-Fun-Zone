# “等位奇遇局”设计方案

## 1 设计思路

### 1.1 项目简介

本项目名为“等位奇遇局”，旨在面向本地生活到店等待场景，核心目标是把用户排队等位时的空白时间，转化为 AI 互动、店铺内容种草、个性化推荐和轻量优惠转化的机会。项目不做泛泛陪聊，而是围绕“等待状态 + 商家知识 + 用户偏好 + AI 互动”设计一条完整体验链路。

**项目的主流程为：**

**用户进入等位状态**
**→ 系统展示当前排队信息和预计等待时间**
**→ 根据等待时长推荐合适的互动模式**
**→ 用户与 AI 进行菜单速看、人格测试、海龟汤或自由聊天**
**→ AI 基于商家知识生成推荐内容**
**→ 系统记录用户偏好和互动结果**
**→ 在特定条件下展示优惠券或套餐推荐**

### 1.2 场景定位

本项目选择“餐饮等位”作为 MVP 主场景，因为排队等待是本地生活中高频、真实且容易产生流失的场景。用户在等待时通常有三类需求：缓解无聊、快速了解店铺、提前完成点单决策。商家则希望利用等待阶段推广招牌菜、套餐和优惠权益，降低等位流失。因此，项目设计重点不是单纯做小游戏，而是把等待时间变成一个可运营的互动窗口。

### 1.3 AI 互动设计

AI 互动围绕商家内容展开，避免脱离店铺场景。

①菜单速看模式中，AI 用简短话术总结本店重点推荐、适合新客的菜品和快速点单组合。

②人格测试模式中，用户回答若干偏好问题，系统计算口味、社交、性价比、尝鲜度等维度分数，并生成对应的美食人格、推荐菜品和推荐理由。

③海龟汤模式中，AI 作为主持人给出谜面，用户通过“是 / 否 / 部分是 / 无关”的方式推理答案。猜中后展示答案解释，并关联店铺特色或商品推荐。

④自由聊天模式中，AI 识别用户等待情绪和点单意向，进行安抚回应，同时推荐适合当前场景的套餐或商品。

### 1.4 RAG 商家知识库设计

项目引入商家知识库，让 AI 内容尽量来自真实商家信息。商家后台支持上传 PDF 或 Word 文档，系统解析后切分为知识片段，并存入知识库。

这些知识可用于生成：

人格测试题目
人格测试结果
店铺海龟汤谜题
菜单推荐话术
商品解释和套餐推荐

这样可以减少 AI 编造不存在商品的问题，也让不同商家的互动内容具备差异化。

### 1.5 商家后台设计

商家后台用于支持 AI 互动内容配置，主要包括三部分：

店铺信息管理：维护商家基础信息。
文档管理：上传菜单、商品介绍、品牌故事等资料，形成知识库。
内容生成：基于知识库生成商品人格测试和店铺海龟汤内容。

后台的作用是让商家不需要写代码，也能配置属于自己店铺的 AI 等位互动活动。

## 2 技术框架

### 2.1 框架一览

前端采用 React + Vite 构建，组件化拆分为等待头部、模式切换、聊天区、推荐区、优惠券区和商家后台模块。

状态管理主要通过自定义 Hooks 实现：

useWaitStatus 管理等待状态、等待时间和模式推荐。
useAIChat 管理 AI 对话、人格测试进度、海龟汤状态和消息流。

数据层使用 Mock 数据和 Supabase 结合。Mock 数据用于演示商品、优惠券和等位状态；Supabase 用于商家信息、文档、知识库片段和生成内容的存储。

### 2.2 框架详细说明

本项目采用 React + Vite 作为前端基础框架，整体定位为一个面向移动端 H5 场景的单页应用。项目入口由 main.jsx 挂载 React 应用，路由层通过 react-router-dom 组织用户端与商家端页面，包括首页入口、用户等位互动页以及商家管理后台。Vite 负责本地开发、构建和预览，Tailwind CSS 负责页面样式体系，Radix UI 组件库与本地封装的 ui 组件共同提供按钮、卡片、弹窗、标签页、进度条等基础交互组件。

前端页面采用组件化架构。用户端核心页面由 Index.jsx 统一编排，页面被拆分为等待状态头部、互动模式切换区、AI 聊天区、推荐商品区、优惠券区和底部导航等模块。其中，WaitHeader 负责展示店铺名称、排队号码、已等待时间、预计等待时间和推荐模式；ModeSwitcher 负责切换菜单速看、人格测试、店铺海龟汤和自由聊天等互动模式；ChatArea 负责承载 AI 消息流、用户输入、人格测试选项、答题进度和海龟汤剩余提问次数；RecommendationArea 和 CouponArea 则分别负责根据互动结果展示商品推荐和可领取优惠券。

项目的数据与业务状态主要通过自定义 React Hooks 管理。useWaitStatus 负责维护等位状态，包括店铺名称、当前号码、已等待时长、预计等待时间和前方排队人数，并根据等待时间推导可用互动模式和推荐模式。useAIChat 负责管理 AI 对话流程，包括消息列表、加载状态、人格测试进度、用户答案、维度分数、海龟汤题目、剩余提问次数和是否猜中等状态。通过这种方式，页面组件只负责展示和触发事件，核心业务流程被集中在 Hooks 中，增强了代码的可维护性。

AI 能力被封装在 services/longcatApi.js 中，形成独立的服务层。该模块负责与大模型接口进行交互，并提供菜单问答、自由聊天、人格测试生成、人格测试结果计算、海龟汤生成和海龟汤问答判断等能力。用户端并不直接关心模型调用细节，而是通过 useAIChat 调用这些服务函数，从而实现前端交互逻辑与 AI 编排逻辑的分离。

商家知识库相关能力由 services/knowledgeBase.js 和 services/documentParser.js 承担。商家后台支持上传 PDF 或 Word 文档，系统解析文档文本后按段落切分为知识片段，并存储到 Supabase 数据表中。后台的 ContentBuilder 会读取这些知识片段，将其作为上下文传入 AI 生成服务，用于生成商品人格测试和店铺海龟汤内容。这一设计相当于实现了轻量级 RAG 流程，即通过商家上传资料形成知识上下文，再约束 AI 围绕真实商家内容生成互动内容。

数据存储层使用 Supabase。项目中通过 integrations/supabase/client.js 初始化 Supabase 客户端，并在知识库服务中操作 merchants、merchant_documents、merchant_kb_chunks 和 generated_content 等数据表。商家基础信息、上传文档、知识库片段和已生成互动内容均可以持久化保存。同时，项目仍保留 mockData.js 作为演示数据来源，用于提供店铺信息、菜单数据、优惠券数据、互动模式配置和初始 AI 消息，保证在没有完整后端数据时也能跑通主要演示链路。

商家端采用独立后台页面实现，入口为 MerchantDashboard.jsx。后台通过标签页组织店铺信息管理、文档管理和内容生成三个模块。ProfileEditor 负责商家基础资料配置，DocumentUploader 负责文档上传、解析、切分和入库，ContentBuilder 负责基于知识库生成互动内容。该后台为用户端 AI 互动提供内容来源，使项目不仅是一个用户侧页面，也具备商家可配置、可运营的基础能力。

优惠券与推荐模块采用轻量规则实现。RecommendationArea 根据当前互动模式和人格测试结果展示不同商品或套餐推荐；CouponArea 根据等待时间、人格测试完成状态和海龟汤猜中状态判断是否展示对应优惠券。这种实现没有引入复杂券控引擎，而是以清晰的前端规则完成 MVP 阶段的“互动结果到优惠转化”闭环。

整体来看，本项目的技术框架可以概括为：React + Vite 构建移动端单页应用，Tailwind CSS 和组件库支撑界面层，React Router 管理页面路由，自定义 Hooks 承载等待状态与 AI 互动流程，服务层封装大模型调用、文档解析和知识库操作，Supabase 提供数据持久化能力，Mock 数据保障演示链路完整。该架构符合 MVP 项目的实现目标，既能快速完成用户与 AI 等位互动功能，也为后续接入真实排队系统、真实优惠券系统和更完整的 RAG 检索能力保留了扩展空间。

## 3 使用说明

### 3.1 商家端

商家进入项目首页后，选择“**商家后台**”入口，进入商家管理中心。商家端主要用于维护店铺信息、上传店铺资料，并生成用户端可使用的 AI 互动内容。

进入后台后，首先在“店铺信息”模块中填写或修改店铺基础资料，例如店铺名称等信息。保存后，这些信息会作为用户端展示和 AI 内容生成的基础数据。

随后进入“文档管理”模块，上传与店铺相关的 Word 或 PDF 文档，例如菜单、商品介绍、套餐说明、服务亮点、品牌故事等。系统会自动解析文档内容，并将文本切分为知识库片段，用于后续 AI 生成互动内容。

完成文档上传后，进入“内容生成”模块。商家可以基于已上传的知识库内容生成“商品人格测试”和“店铺海龟汤”等互动内容。生成后的内容会保存为当前商家的可用互动素材，供用户端在等位互动过程中调用。

**商家端的基本使用流程为：**

**进入商家后台**
**→ 编辑店铺信息**
**→ 上传菜单或店铺资料**
**→ 系统解析并生成知识库**
**→ 生成商品人格测试和店铺海龟汤内容**
**→ 用户端使用这些内容进行 AI 等位互动**

### 3.2 用户端

用户进入项目首页后，选择“用户端”入口，进入 AI 等位互动页面。页面顶部会展示当前店铺名称、排队号码、前方排队人数、已等待时间和预计剩余等待时间，帮助用户了解当前等位状态。

进入页面后，系统会根据等待状态展示推荐互动模式。用户也可以手动切换不同互动方式，包括菜单速看、商品人格测试、店铺海龟汤和自由聊天。

在“菜单速看”模式下，用户可以直接向 AI 询问店铺招牌、推荐菜品、套餐搭配或点单建议。AI 会围绕当前店铺内容进行回答，帮助用户快速了解菜单。

在“商品人格测试”模式下，用户根据页面显示的问题选择答案。完成测试后，系统会生成用户的美食人格结果，并展示对应的推荐菜品、推荐理由和专属优惠权益。

在“店铺海龟汤”模式下，用户根据 AI 给出的谜面进行提问和猜测。用户可以通过输入问题逐步推理答案，系统会显示剩余提问次数。猜中答案或游戏结束后，页面会展示答案解释，并可能出现对应的互动奖励券。

在“自由聊天”模式下，用户可以直接和 AI 对话，询问菜单、套餐、店铺特色，也可以表达等待感受。AI 会根据用户消息进行安抚、推荐和引导。

**用户端的基本使用流程为：**

**进入用户端**
**→ 查看当前等位状态**
**→ 选择或接受推荐互动模式**
**→ 与 AI 进行菜单问答、人格测试、海龟汤或自由聊天**
**→ 查看推荐商品或套餐**
**→ 满足条件后领取优惠券**
